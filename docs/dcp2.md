## Часть I. Концепция и принципы работы DCP2

<a id="toc-002"></a>
### 1. Назначение протокола

DCP2 — это прикладной протокол обмена между внешним клиентом и устройством по TCP. Его задача — дать единый, предсказуемый и удобный способ передавать команды, получать результаты и принимать асинхронные уведомления в рамках одного соединения.
 
Протокол спроектирован так, чтобы оставаться простым для реализации и расширяемым по мере развития системы. Общий формат сообщений сохраняет целостность взаимодействия между компонентами, а разделение обмена на сервисы помогает поддерживать понятную структуру и масштабируемость.

Основной принцип DCP2 — единые правила работы для всех сценариев обмена. Это снижает сложность клиентской и серверной логики, упрощает сопровождение и делает поведение протокола устойчивым при добавлении новых возможностей.

Документ ниже последовательно описывает структуру сообщений, назначение полей и форматы данных, используемые в рамках DCP2.

<a id="toc-003"></a>
### 2. Архитектура протокола

DCP2 работает как прикладной слой поверх установленного TCP-соединения между клиентом и устройством.
На уровне транспорта данные поступают как непрерывный байтовый поток, поэтому границы логических сообщений определяются средствами самого DCP2.

Каждый кадр DCP2 состоит из двух логических частей:

`DCP2_FRAME = DCP2_HDR + DCP2_PAYLOAD`

<a id="toc-004"></a>
#### 2.1. Заголовок фрейма `DCP2_HDR`

Загловок `DCP2_HDR` решает транспортно-прикладную задачу фрейминга сообщений в TCP-потоке.
Поскольку TCP не сохраняет границы сообщений, именно заголовок позволяет корректно определить начало кадра, проверить его базовую корректность и понять, где заканчивается текущий кадр.

На практике `DCP2_HDR` используется для:
синхронизации и выделения границ сообщения в непрерывном потоке, первичной валидации входного кадра до передачи в сервисную логику, контроля совместимости версии протокола и определения длины полезной части кадра для полного чтения `DCP2_PAYLOAD`.

`DCP2_HDR` отделяет задачу надёжного разбора потока от прикладной обработки команд.

<a id="toc-005"></a>
#### 2.2. Полезная нагрузка фрейма `DCP2_PAYLOAD`

`DCP2_PAYLOAD` описывает смысл сообщения после того, как кадр выделен и принят как валидный.
Эта часть отвечает за маршрутизацию в нужный сервис и за интерпретацию конкретной операции.

Логика обработки `DCP2_PAYLOAD` последовательна: `srv` определяет целевой сервис-обработчик, `message.op` задаёт тип сообщения (`request`, `response`, `event`) и код операции, `message.seq` обеспечивает корреляцию между запросами и ответами, а `message.body` разбирается по контексту `srv + тип + opcode`.

Такой подход даёт единый каркас обмена для всех сервисов: внешняя форма сообщения одинакова, а внутренняя структура тела выбирается контекстно и может расширяться без изменения базовой модели протокола.
Точные форматы полей и допустимые значения приводятся в части II (формальная спецификация).

<a id="toc-006"></a>
#### 2.3. Карта структур DCP2

```text
TCP_APP_DATA (byte stream)
└── DCP2_FRAME
    ├── DCP2_HDR
    │   ├── magic[4]
    │   ├── ver
    │   └── dcp_len
    └── DCP2_PAYLOAD
        ├── srv (дискриминатор сервиса)
        │   ├── 0x00 -> PING
        │   ├── 0x01 -> MEM
        │   ├── 0x02 -> I2C
        │   ├── 0x03 -> SMI
        │   ├── 0x04 -> SPI
        │   ├── 0x05 -> UART
        │   ├── 0x06 -> NOTIFY
        │   └── 0x7F -> VENDOR
        └── SRV_MESSAGE (формат зависит от srv)
            ├── op   (RESP/EVENT/OPCODE)
            │   └── OPCODE (зависит от srv):
            │       ├── srv=PING (0x00)
            │       │   └── 0x00 PING
            │       │       ├── request  -> PING_REQUEST
            │       │       └── response -> PING_RESPONSE
            │       ├── srv=MEM (0x01)
            │       │   ├── 0x00 MEM_READ
            │       │   │   ├── request  -> MEM_READ_REQUEST
            │       │   │   └── response -> MEM_READ_RESPONSE
            │       │   └── 0x01 MEM_WRITE
            │       │       ├── request  -> MEM_WRITE_REQUEST
            │       │       └── response -> MEM_WRITE_RESPONSE
            │       ├── srv=I2C (0x02)
            │       │   ├── 0x00 I2C_READ_REG
            │       │   │   ├── request  -> I2C_READ_REG_REQUEST
            │       │   │   └── response -> I2C_READ_REG_RESPONSE
            │       │   ├── 0x01 I2C_WRITE_REG
            │       │   │   ├── request  -> I2C_WRITE_REG_REQUEST
            │       │   │   └── response -> I2C_WRITE_REG_RESPONSE
            │       │   └── 0x02 I2C_POLICY_SET
            │       │       ├── request  -> I2C_POLICY_SET_REQUEST
            │       │       └── response -> I2C_POLICY_SET_RESPONSE
            │       ├── srv=SMI (0x03)
            │       │   ├── 0x00 SMI_READ
            │       │   │   ├── request  -> SMI_READ_REQUEST
            │       │   │   └── response -> SMI_READ_RESPONSE
            │       │   └── 0x01 SMI_WRITE
            │       │       ├── request  -> SMI_WRITE_REQUEST
            │       │       └── response -> SMI_WRITE_RESPONSE
            │       ├── srv=NOTIFY (0x06)
            │       │   ├── 0x10 SUBSCRIBE
            │       │   │   ├── request  -> NOTIFY_SUBSCRIBE_REQUEST
            │       │   │   ├── response -> NOTIFY_SUBSCRIBE_RESPONSE
            │       │   │   └── event    -> NOTIFY_EVENT
            │       │   └── 0x11 UNSUBSCRIBE
            │       │       ├── request  -> NOTIFY_UNSUBSCRIBE_REQUEST
            │       │       └── response -> NOTIFY_UNSUBSCRIBE_RESPONSE
            │       └── srv=PL (I2C/SMI/SPI/UART)
            │           ├── 0x10 SUBSCRIBE_STREAM
            │           │   ├── request  -> PL_SUBSCRIBE_STREAM_REQUEST
            │           │   ├── response -> PL_SUBSCRIBE_STREAM_RESPONSE
            │           │   └── event    -> PL_STREAM_EVENT
            │           └── 0x11 UNSUBSCRIBE_STREAM
            │               ├── request  -> PL_UNSUBSCRIBE_STREAM_REQUEST
            │               └── response -> PL_UNSUBSCRIBE_STREAM_RESPONSE
            ├── seq
            └── body (определяется по RESP/EVENT и srv/opcode)
                ├── request  -> SRV_REQUEST_BODY
                ├── response -> SRV_RESPONSE_BODY
                └── event    -> SRV_EVENT_BODY
```


<a id="toc-007"></a>
### 3. Модель обмена
<a id="toc-008"></a>
#### 3.1. Участники и роли

Обмен строится между двумя сторонами: внешним клиентом и сервером на устройстве. Клиент инициирует управляющие операции и ожидает результат, сервер принимает сообщения, маршрутизирует их по сервисам и возвращает ответы. Асинхронные события инициируются сервером.

<a id="toc-009"></a>
#### 3.2. Типы сообщений

В протоколе используются три типа сообщений:

| Тип | Инициатор | Назначение |
|---|---|---|
| `request` | Клиент | Запрос на выполнение операции |
| `response` | Сервер | Результат обработки запроса |
| `event` | Сервер | Асинхронное сообщение без отдельного запроса в момент отправки |

```text
Клиент                          Сервер
  |                               |
  |-------- request ------------->|
  |<------- response -------------|
  |                               |
  |<======== event ===============|   (асинхронно, по инициативе сервера)
```

`request` используется для инициирования действия (чтение-запись, изменение политики, управление подпиской), `response` является обязательным итогом обработки `request`, а `event` передаётся сервером по собственной инициативе при появлении нового состояния или данных (например, в активной подписке на стрим).

<a id="toc-010"></a>
#### 3.3. Корреляция сообщений

`seq` — это идентификатор логической операции в рамках одного TCP-соединения.
Он нужен для того, чтобы клиент мог однозначно сопоставить полученный `response` с конкретным ранее отправленным `request`, даже если в канале одновременно выполняется несколько операций.

Правила использования `seq`:

| Контекст   | Требование                                                          |
| ---------- | ------------------------------------------------------------------- |
| `request`  | Значение `seq` назначает клиент в диапазоне `1..0xFFFF`             |
| `response` | Сервер возвращает то же `seq`, что было в соответствующем `request` |
| `event`    | Используется `seq = 0`                                              |

Рекомендуемая практика на стороне клиента: использовать монотонно возрастающий счётчик `seq` с переходом через `0xFFFF` по кругу, не переиспользовать номер до получения итогового `response`, вести таблицу активных запросов по ключу `seq` и не применять `seq = 0` для обычных запросов.

Пример корреляции при параллельных запросах:

Клиент отправляет три запроса: `MEM_READ(seq=0x0101)`, `I2C_READ_REG(seq=0x0102)`, `SMI_READ(seq=0x0103)`. Сервер может вернуть ответы в другом порядке, например: `response(seq=0x0103)`, затем `response(seq=0x0101)`, затем `response(seq=0x0102)`. Клиент сопоставляет каждый ответ по `seq` с таблицей активных запросов, а не по порядку получения, после чего соответствующий запрос помечается завершённым независимо от позиции ответа в потоке.

<a id="toc-011"></a>
#### 3.4. Модель ошибок

Результат операции интерпретируется в два этапа: сначала анализируется общий `status`, затем (только при успешном выполнении) разбирается тело ответа. Полный перечень кодов и их формальные значения приведён в разделе `2.2.4.1. Поле status`; в данном разделе фиксируется практическое поведение сторон протокола.

Сервер при ошибке возвращает `response` с соответствующим кодом `status` и завершает обработку текущего запроса. Ошибка не должна нарушать обработку других независимых запросов в том же соединении.

Клиент при получении ошибки считает текущую операцию завершённой с отказом, выполняет действие по классу ошибки и продолжает работу соединения, если транспорт остаётся валидным.

| Класс ошибки | Типичные коды | Действие клиента |
|---|---|---|
| Ошибка формата или протокола | `ERR_MALFORMED`, `ERR_UNSUPPORTED` | Считать запрос некорректным, не отправлять повторно в том же виде, исправить параметры |
| Временная ошибка ресурса | `ERR_BUSY`, `ERR_TIMEOUT` | Повторить запрос после задержки с ограничением числа попыток |
| Ошибка доступа или политики | `ERR_DENIED` | Не повторять запрос до изменения политики, прав или режима работы |
| Ошибка параметров запроса | `ERR_RANGE` | Скорректировать параметры и отправить новый корректный запрос |
| Внутренняя ошибка сервиса | `ERR_INTERNAL` | Допустить ограниченное число повторных попыток, затем зафиксировать ошибку и уведомить оператора/журнал |

<a id="toc-012"></a>
#### 3.5. Типовые сценарии обмена

Ниже приведены базовые сценарии, по которым разбирается и обрабатывается входящий фрейм `DCP2`.

<a id="toc-013"></a>
##### 3.5.1. Корректный `request` с успешным ответом

Пример: `srv=MEM`, `op=request/MEM_READ`, `seq=0x0101`.

Кадр, обрабатываемый сервером (hex):

```text
44 43 50 32 | 00 02 | 00 0C | 01 | 00 | 01 01 | 00 20 40 00 00 00 00 02
magic       | ver   | dcp_len|srv| op | seq   | MEM_READ_REQUEST
```

Семантика полей:

| Поле                     | Значение                                    |
| ------------------------ | ------------------------------------------- |
| `DCP2_HDR.magic`         | `44 43 50 32` (`"DCP2"`)                    |
| `DCP2_HDR.ver`           | `00 02`                                     |
| `DCP2_HDR.dcp_len`       | `00 0C`                                     |
| `DCP2_PAYLOAD.srv`       | `01` (`MEM`)                                |
| `SRV_MESSAGE.op`         | `00` (`request`, `OPCODE=0x00`, `MEM_READ`) |
| `SRV_MESSAGE.seq`        | `01 01` (`0x0101`)                          |
| `MEM_READ_REQUEST.flags` | `00`                                        |
| `MEM_READ_REQUEST.width` | `20` (`32` бита)                            |
| `MEM_READ_REQUEST.addr`  | `40 00 00 00`                               |
| `MEM_READ_REQUEST.count` | `00 02`                                     |

Кадр, обрабатываемый клиентом (hex):

```text
44 43 50 32 | 00 02 | 00 0A | 01 | 80 | 01 01 | 00 00 | DE AD BE EF
magic       | ver   | dcp_len|srv| op | seq   |status | data
```

Семантика полей:

| Поле                     | Значение                                     |
| ------------------------ | -------------------------------------------- |
| `DCP2_HDR.magic`         | `44 43 50 32` (`"DCP2"`)                     |

### 4. Расширяемость и совместимость

`DCP2` ориентирован на эволюцию без изменения базового каркаса обмена: новые возможности добавляются через новые сервисы и операции, а действующие сценарии сохраняют предсказуемое поведение.

На концептуальном уровне целевое правило простое: расширение протокола не должно ломать ранее реализованных клиентов, если они не используют новые возможности.

<a id="toc-018"></a>
### 5. Нормативные правила интерпретации

<a id="toc-019"></a>
#### 5.1. Порядок байтов

Все поля размером более 1 байта (`u16`, `u32`, `u64`) передаются в сетевом порядке байтов (big-endian). Это правило едино для заголовка, полезной нагрузки, кодов состояния и полей событий.

<a id="toc-020"></a>
#### 5.2. Поведение при ошибках фрейминга

Если входящий кадр не проходит базовую проверку (`magic`, `ver`, `dcp_len`), сервер отклоняет кадр как некорректный. Поведение соединения после отклонения определяется реализацией: допускается продолжение приёма следующего кадра или закрытие соединения при устойчивой десинхронизации потока.

<a id="toc-021"></a>
#### 5.3. Глобальные ограничения длины

Реализация обязана проверять длины всех переменных полей до начала выполнения операции. При несоответствии длины фактическому формату сообщения должен возвращаться `ERR_MALFORMED`, а при превышении допустимых значений параметров — `ERR_RANGE`.

Пример: клиент отправил `MEM_WRITE` с `count=2`, `width=32`, но передал только 2 байта `data` вместо ожидаемых 8. Такая длина тела не соответствует формату `MEM_WRITE_REQUEST`, поэтому сервер должен отклонить запрос и вернуть `response` со статусом `ERR_MALFORMED`.

<a id="toc-022"></a>
#### 5.4. Правила использования `seq`

`seq` используется как ключ корреляции `request`/`response` в пределах одного TCP-соединения и применяется по следующим правилам:

1. Клиент назначает `seq` в диапазоне `0x0001..0xFFFF`; значение `0x0000` для `request` не используется (зарезервировано для `event`).
2. Клиент обязан вести таблицу активных запросов (ключ — `seq`) для корреляции `response`, контроля таймаутов и безопасного переиспользования номеров.
3. Новый `request` может получить только свободный `seq`, которого нет в таблице активных запросов.
4. `seq` считается освобождённым после перехода операции в терминальное состояние: получен валидный `response` с тем же `seq` или запрос закрыт локально по таймауту/отмене.
5. При переполнении счётчика клиент продолжает выдачу по кругу (`...0xFFFE`, `0xFFFF`, `0x0001`), пропуская занятые значения.
6. `response` с неизвестным или устаревшим `seq` не должен нарушать обработку активных запросов; такие сообщения обрабатываются по политике клиента (например, логирование и игнорирование).

<a id="toc-023"></a>
#### 5.5. Таймауты и повторные попытки

Клиент применяет таймаут к каждому активному `request` и ведёт счётчик попыток.

Базовые параметры по умолчанию:

| Параметр | Значение |
|---|---|
| `T_resp` (ожидание `response`) | `1000 ms` |
| `N_retry_max` | `2` повторные попытки (итого до 3 отправок) |

`T_resp` применяется одинаково для всех типов `request` в рамках текущей версии протокола.

Правила обработки:

1. При отправке `request(seq=N)` клиент создаёт запись в таблице активных запросов: `seq`, тип операции, время отправки, дедлайн, номер попытки.
2. Если до дедлайна получен валидный `response(seq=N)`, запрос завершается успешно или ошибкой по `status`, `seq` освобождается.
3. Если дедлайн истёк, запрос закрывается локально с признаком timeout, текущий `seq` освобождается.
4. Повторная отправка выполняется только как новый `request` с новым `seq`; повторно использовать старый `seq` для retry недопустимо.
5. Повторные попытки выполняются только для временных условий (`ERR_BUSY`, `ERR_TIMEOUT`, отсутствие ответа). Для `ERR_MALFORMED`, `ERR_RANGE`, `ERR_DENIED`, `ERR_UNSUPPORTED` повтор без изменения параметров не выполняется.
6. После достижения `N_retry_max` операция завершается ошибкой timeout на стороне клиента.
7. Поздний `response` по уже закрытому запросу (устаревший `seq`) не влияет на состояние активных операций: такой кадр логируется и игнорируется.

<a id="toc-023a"></a>
#### 5.6. Обнаружение возможностей сервера

В текущей версии протокола отдельная операция `CAPABILITIES` не используется. Клиент ориентируется на фиксированный контракт версии протокола и проверяет поддержку операций по факту ответа сервера.

Если операция или сервис не поддерживаются, сервер возвращает `ERR_UNSUPPORTED`, после чего клиент выбирает совместимый сценарий или отключает соответствующую функцию.

Операция `CAPABILITIES` может быть добавлена в будущих версиях как опциональное расширение, если потребуется динамическое согласование возможностей.

<a id="toc-023b"></a>
#### 5.7. Политика версионирования

Поле `ver` в `DCP2_HDR` определяет версию базового wire-контракта протокола.

Таблица версий:

| `ver` | Статус | Содержание |
|---|---|---|
| `0x0001` | legacy baseline | Базовый каркас DCP2: `PING`, `MEM`, `I2C`, `SMI`, `SPI`, `UART`, `VENDOR` |
| `0x0002` | current | Добавлена формальная политика версионирования и сервис `NOTIFY` для асинхронных control/state уведомлений |

Нормативные правила:

1. Сервер обязан проверять `ver` до обработки `DCP2_PAYLOAD` и отклонять неподдерживаемую версию кадра.
2. Клиент обязан трактовать `ver` как основной признак совместимости wire-формата.
3. Изменения документации без изменения бинарного формата и семантики кадров не меняют `ver`.
4. Обратно-совместимые расширения могут сохранять `ver`, только если не меняют формат и семантику уже существующих `srv`, `OPCODE`, `status` и обязательных полей.
5. Изменения, затрагивающие формат `DCP2_HDR`, существующие `body`-структуры, семантику существующих полей или вводящие новый обязательный для корректной совместимости сервис, обязаны повышать `ver`.
6. Для версии `0x0002` сервис `NOTIFY` считается частью формального контракта протокола.

## Часть II. Формальная спецификация DCP2

<a id="toc-025"></a>
## 1. Структура DCP2_HDR

```c
typedef struct {
    u8  magic[4];
    u16 ver;
    u16 dcp_len;
} DCP2_HDR;         

```

**Размер**: 8 байт
**Назначение**: Предназначен для фрейминга, синхронизации и контроля версии сообщений DCP2 в потоковом транспорте TCP

<a id="toc-026"></a>
### 1.1. Поле `magic`

**Размер**: 4 байта
**Назначение**: Сигнатура протокола. Имеет фиксированное значение в ASCII "DCP2" или в шестнадцатеричном виде `0x44 0x43 0x50 0x32`
**Пример**: `44 43 50 32`

<a id="toc-027"></a>
### 1.2. Поле `ver`

**Размер**: 2 байта
**Назначение**: Версия wire-контракта протокола DCP2.
**Допустимые значения**:
- `0x0001` — legacy baseline
- `0x0002` — текущая версия документа
**Пример**: `00 02`

<a id="toc-028"></a>
### 1.3. Поле `dcp_len`

**Размер**: 2 байта
**Назначение**: Длина поля `DCP2_PAYLOAD` в байтах
**Пример**: `00 04`

<a id="toc-029"></a>
## 2. Структура `DCP2_PAYLOAD`

```c
typedef struct {
    u8          srv;
    SRV_MESSAGE message;
} DCP2_PAYLOAD;
```

**Размер**: `DCP_LEN` байт
**Назначение**:  Предназначена для инкапсуляции одного логического сообщения DCP2, мультиплексирования сервисов внутри протокола и передачи данных конкретной подсистеме (PING, MEM, I2C и т.д.)

<a id="toc-030"></a>
### 2.1. Поле `srv`

**Размер**: 1 байт
**Назначение**:  Идентификатор сервиса (дискриминатор), определяющий формат и семантику `service_message`.

Таблица допустимых значений:

|Значение|Сервис|
|---|---|
|0x00|PING|
|0x01|MEM|
|0x02|I2C|
|0x03|SMI|
|0x04|SPI|
|0x05|UART|
|0x06|NOTIFY|
|0x7F|VENDOR|

<a id="toc-031"></a>
### 2.2. Структура `SRV_MESSAGE`

```c
typedef struct {
    u8  op;
    u16 seq;

    /* тип body определяется битами RESP/EVENT в op */

    union {
        SRV_REQUEST_BODY  request;
        SRV_RESPONSE_BODY response;
        SRV_EVENT_BODY    event;
    } body;

} SRV_MESSAGE;

```

**Размер**: `DCP_LEN - 1` байт (в составе `DCP2_PAYLOAD`)
**Назначение**:  Содержит одно логическое сообщение выбранного сервиса (запрос, ответ или асинхронное событие).
**Примечание**: 
1. Все сообщения сервиса начинаются с общего заголовка (`op`, `seq`), обеспечивающего:
	- унифицированную корреляцию request/response 
	- поддержку асинхронных событий
	- единый механизм обработки ошибок
2. Формат тела сообщения определяется значением `srv`, типом сообщения (битами `RESP`/`EVENT`) и `OPCODE` (bit5..0 поля `op`).
3.  `union` приведён только для пояснения формата.  В реальном потоке данных после поля `seq` сразу идут поля сообщения, длина которых зависит от типа сообщения.

<a id="toc-032"></a>
#### 2.2.1. Поле `op`

**Размер**: 1 байт
**Назначение**:  Код операции внутри сервиса и тип сообщения.

Битовое представление:
- `bit7` — `RESP` (0=request, 1=response)
- `bit6` — `EVENT` (0=обычное сообщение, 1=асинхронное событие)
- `bit5..0` — `OPCODE` (0..63)

<a id="toc-033"></a>
#### 2.2.2. Поле `seq`

**Размер**: 2 байта
**Назначение**:  Идентификатор сообщения для корреляции request/response.
- В запросе задаётся клиентом в диапазоне `1..0xFFFF`.
- В ответе сервер повторяет значение `seq`.
- Значение `seq = 0` зарезервировано для асинхронных событий (`event`).

<a id="toc-034"></a>
#### 2.2.3. Структура `SRV_REQUEST_BODY`

```c
typedef struct {
    union {
        PING_REQUEST ping;
        
        MEM_READ_REQUEST  mem_read;
        MEM_WRITE_REQUEST mem_write;
        
        I2C_READ_REG_REQUEST    i2c_read_reg;
        I2C_WRITE_REG_REQUEST   i2c_write_reg;
        I2C_POLICY_SET_REQUEST  i2c_policy_set;

        SMI_READ_REQUEST  smi_read;
        SMI_WRITE_REQUEST smi_write;

        NOTIFY_SUBSCRIBE_REQUEST   notify_subscribe;
        NOTIFY_UNSUBSCRIBE_REQUEST notify_unsubscribe;

        PL_SUBSCRIBE_STREAM_REQUEST   pl_subscribe_stream;
        PL_UNSUBSCRIBE_STREAM_REQUEST pl_unsubscribe_stream;

        VENDOR_REQUEST vendor;

    } u;
} SRV_REQUEST_BODY;
```

**Размер**: переменный; определяется значениями `srv` и `OPCODE` (bit5..0 поля `op`)
**Назначение**:  Содержит тело запроса (request) для выбранного сервиса и операции

<a id="toc-035"></a>
##### 2.2.3.1 Структура `PING_REQUEST`

```c
typedef struct {
    /* пусто */
} PING_REQUEST;
```

**Размер**: 0 байт
**Назначение**:  Используется для проверки доступности соединения и базовой работоспособности сервера.
**Примечание**: Структура `PING_REQUEST` не содержит дополнительных полей. Запрос состоит только из общего заголовка `SRV_MESSAGE`

<a id="toc-036"></a>
##### 2.2.3.2 Структура `MEM_READ_REQUEST`

```c
typedef struct {
    u8  flags;
    u8  width;
    u32 addr;
    u16 count;
} MEM_READ_REQUEST;
```

**Размер**: 8 байт
**Назначение**:  Определяет параметры операции чтения памяти (MMIO/AXI адресного пространства).
**Примечание**: Контекст использования — `srv = 0x01` (`MEM`), `RESP = 0`, `EVENT = 0`, `OPCODE = 0x00`.

<a id="toc-037"></a>
###### 2.2.3.2.1. Поле `flags`

**Размер**: 1 байт
**Назначение**:  Определяет режим выполнения операции чтения.

Битовое представление:
- `bit0` — `AUTOINC`
    - `0` — чтение всех элементов из одного и того же адреса
    - `1` — после каждого чтения адрес увеличивается на `width / 8`
- `bit7..1` — зарезервированы (должны быть равны 0)

**Примечание**: Если зарезервированные биты установлены → `ERR_MALFORMED`.

<a id="toc-038"></a>
###### 2.2.3.2.2. Поле `width`

**Размер**: 1 байт  
**Назначение**: Определяет разрядность одного читаемого элемента в битах.

Допустимые значения:
- `8`
- `16`
- `32`
- `64`

**Примечание**:
- Если значение не поддерживается реализацией → `ERR_UNSUPPORTED`.
- При `AUTOINC = 1` шаг увеличения адреса равен `width / 8` байт.

<a id="toc-039"></a>
###### 2.2.3.2.3. Поле `addr`

**Размер**: 4 байта  
**Назначение**: Базовый адрес чтения в адресном пространстве памяти (MMIO/AXI).

**Примечание**:
- Интерпретация адреса определяется реализацией.
- Если адрес выходит за разрешённый диапазон → `ERR_RANGE`.

<a id="toc-040"></a>
##### 2.2.3.3. Структура `MEM_WRITE_REQUEST`

```c
typedef struct {
    u8  flags;
    u8  width;
    u32 addr;
    u16 count;
    u8  data[];   /* длина = count * (width / 8) */
```

##### 2.2.5.2. Структура `PL_STREAM_EVENT`

```c

    u32 lost_delta;
    u16 data_len;
    u8  data[];
} PL_STREAM_EVENT;
```

**Размер**: `1 + 4 + 2 + data_len` байт без `time_us`; `9 + 4 + 2 + data_len` байт с `time_us`
**Назначение**: Передаёт порцию данных стрима выбранного PL-сервиса (`I2C`, `SMI`, `SPI`, `UART`) и диагностику потерь.

<a id="toc-066"></a>
###### 2.2.5.2.1. Поле `ev_flags`

**Размер**: 1 байт
**Назначение**: Флаги состояния стрима для текущего события.

Битовое представление:
- `bit0` — `OVERFLOW` (обнаружена потеря данных с прошлого события)
- `bit7..1` — зарезервированы (должны быть равны 0)

<a id="toc-067"></a>
###### 2.2.5.2.2. Поле `lost_delta`

**Размер**: 4 байта
**Назначение**: Количество потерянных единиц данных с момента предыдущего события (или с момента подписки/сброса счётчиков).

<a id="toc-068"></a>
###### 2.2.5.2.3. Поле `data_len`

**Размер**: 2 байта
**Назначение**: Длина поля `data` в байтах.

**Примечание**: При использовании режима `RAW_WORDS` значение `data_len` должно быть кратно 4.

##### 2.2.5.3. Структура `VENDOR_EVENT`

```c
typedef struct {
    u16 len;
    u8  data[];
} VENDOR_EVENT;
```

**Размер**: `2 + len` байт
**Назначение**: Универсальный контейнер для вендор-специфичных асинхронных событий.
