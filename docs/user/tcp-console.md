# TCP-консоль (порт 8888)

TCP-консоль это основной текстовый пользовательский интерфейс устройства. В текущей прошивке она поднимается как отдельный TCP-сервис на порту `8888` и даёт интерактивную инженерную сессию: команды вводятся строками, ответы приходят как обычный текст, а после каждой команды печатается новый prompt.

Если выбирать один интерфейс, с которого удобнее всего начинать знакомство с живой системой, это почти всегда именно TCP-консоль. Через неё быстрее всего проверить сеть, файловые системы, конфиги и базовое состояние прикладных подсистем.

## Как подключаться

Практически консоль рассчитана на обычный `telnet`-подобный сценарий. Подойдут и `telnet`, и `nc`.

```sh
telnet <device-ip> 8888
```

или

```sh
nc <device-ip> 8888
```

После подключения устройство печатает ASCII-banner, краткую таблицу доступных утилит и prompt. Prompt строится из активного файлового тома и текущего каталога и обычно выглядит как `Zynq/flash>` или `Zynq/flash:config>`.

## Как устроена сессия

У консоли текущей прошивки есть несколько важных свойств, которые полезно понимать заранее.

| Особенность | Что это означает на практике |
|---|---|
| одна команда это одна строка | команда выполняется после `Enter` |
| сравнение команд case-insensitive | `help`, `HELP` и `HeLp` воспринимаются одинаково |
| prompt зависит от файловой сессии | переходы по `cd`, `cd flash`, `cd sd` меняют вид prompt |
| соединение интерактивное, а не пакетное | это не request/response API в духе HTTP |
| сервер слушает с backlog `1` | консоль лучше воспринимать как одиночную инженерную сессию, а не как многопользовательский shell |

Отдельно важно помнить, что консоль живёт поверх текущей сетевой конфигурации. Если вы меняете IP через `ip addr set ...`, текущая TCP-сессия может оборваться сразу после применения адреса.

## Что поддерживается в плане удобства

Текущий консольный цикл не ограничивается “голым вводом строки”. В коде есть базовая telnet-negotiation, история команд, движение курсора и tab-completion.

| Возможность | Статус |
|---|---|
| backspace | работает |
| стрелки влево и вправо | работают |
| история по вверх и вниз | работает |
| tab-completion | работает для команд, части подкоманд и файловых путей |
| `quit` / `exit` | корректно закрывают сессию |

Tab-completion особенно полезен в двух местах. Во-первых, для верхнеуровневых команд вроде `fs`, `ip`, `i2c`, `smi`, `spi`, `mem`, `tar`, `help`. Во-вторых, для некоторых аргументов, например для имён I2C- и SMI-устройств из `config_store`, части подкоманд и файловых путей.

## Что происходит после выполнения команды

Большинство команд в текущей консоли придерживаются простой схемы. Если операция прошла, вы увидите либо `OK`, либо содержательный текстовый вывод. Если что-то пошло не так, обычно приходит `ERR`, иногда с пояснением вроде `ERR: confirm required ...`, `ERR (FR=...)`, `DENIED` или `FS not ready`.

Это важно для автоматизации. Консоль не даёт строгий машинный контракт как DCP2, но для простых скриптов вполне можно ориентироваться на сочетание текстового ответа и повторного появления prompt.

## Верхний уровень команд

На верхнем уровне консоль знает несколько встроенных команд и набор утилит.

| Команда | Что делает |
|---|---|
| `help` | показывает список доступных утилит |
| `reboot` | инициирует reboot через watchdog |
| `quit` / `exit` | закрывает сессию |
| `fs` и файловые команды | работа с `sd:/` и `flash:/` |
| `tar` | создание, просмотр и распаковка tar |
| `ip` | просмотр и изменение сетевой конфигурации |
| `i2c` | работа с I2C-устройствами |
| `smi` | работа с MDIO/SMI |
| `spi` | работа с SPI runtime-shell |
| `mem` | прямой доступ к памяти и MMIO |

Практически это значит, что консоль уже является не “одной командой shell”, а общим инженерным фасадом почти ко всем важным подсистемам устройства.

## Наиболее полезные команды в начале работы

Если устройство только что загружено и нужно быстро понять, живо ли оно, обычно хватает короткого набора команд.

```text
help
ip addr show
pwd
ls
ls /config
ls /www
```

Эти шаги быстро проверяют сразу несколько вещей: работает ли консоль вообще, поднята ли сеть, доступен ли активный файловый том, виден ли `flash:/config` и присутствует ли web UI в `flash:/www`.

## Файловая часть консоли

Файловые команды доступны напрямую, без обязательного префикса `fs`. То есть `pwd`, `ls`, `cd`, `mkdir`, `touch`, `cat`, `rm`, `cp` и `mv` считаются обычными top-level командами консоли.

| Команда | Что делает |
|---|---|
| `pwd` | печатает текущий каталог |
| `ls [path]` | показывает содержимое каталога |
| `cd <dir>` | меняет текущий каталог |
| `cd flash` / `cd sd` | переключает активный том |
| `mkdir <dir>` | создаёт каталог |
| `touch <file>` | создаёт пустой файл |
| `cat <file>` | печатает содержимое файла |
| `rm <path>` | удаляет файл или пустой каталог |
| `rm -r <dir>` | удаляет каталог рекурсивно |
| `cp <src> <dst>` | копирует файл |
| `cp -r <src> <dst>` | копирует каталог рекурсивно |
| `mv <src> <dst>` | перемещает или переименовывает |

Пути можно задавать как относительные, так и явные через `sd:/` и `flash:/`. Поэтому обе формы нормальны:

```text
cd flash
cat /config/network.json
cp flash:/config/network.json sd:/backup/network.json
```

Если нужна краткая подсказка именно по файловому слою, можно вызвать `fs -h`.

## Сеть через `ip`

Команда `ip` это упрощённая оболочка поверх текущего `netif` и `config_store`. Она не повторяет весь `iproute2`, но закрывает базовые сетевые сценарии.

| Команда | Что делает |
|---|---|
| `ip addr show` | показывает текущий IP, prefix, netmask и gateway |
| `ip addr set <IPv4>/<prefix>` | меняет IP и маску, сразу применяет и сохраняет |
| `ip link show` | показывает MAC |
| `ip link set address <mac>` | меняет MAC, сразу применяет и сохраняет |
| `ip route show` | показывает default gateway |
| `ip route set default via <gw>` | меняет gateway, сразу применяет и сохраняет |
| `ip save` | сохраняет текущее runtime-состояние интерфейса в `flash:/config/network.json` |

Здесь важен один практический нюанс. В текущем коде команды `ip ... set` уже сами пытаются и применить, и сохранить конфиг. Поэтому `ip save` нужен не как обязательный финальный шаг после каждого изменения, а скорее как способ зафиксировать текущее runtime-состояние без новой правки параметров.

## Утилиты для работы с железом

Помимо файлов и сети в консоли есть инженерные команды для доступа к аппаратным подсистемам.

| Утилита | Когда она реально полезна |
|---|---|
| `i2c` | активная и штатная часть текущего runtime |
| `spi` | работа с runtime-конфигом и передачей слов в SPI shell |
| `mem` | прямое чтение и запись памяти и MMIO |
| `tar` | упаковка и распаковка каталогов |
| `smi` | код и shell есть, но сама SMI-подсистема сейчас не стартует из стандартного `main()` |

На практике это означает, что `i2c` и `spi` в текущей системе являются нормальными живыми инструментами, а к `smi` лучше относиться осторожно: наличие shell-команды ещё не означает, что вся runtime-цепочка активна после обычной загрузки.

## `i2c` policy

У `i2c` в текущем shell политика доступа теперь оформлена как отдельный под-контекст `policy`. Это важный практический момент: просмотр текущей политики, просмотр списков правил и их редактирование больше не смешаны с короткими legacy-командами `allow`, `deny`, `rules` и `save`.

Базовые команды выглядят так:

| Команда | Что делает |
|---|---|
| `i2c <dev> policy` | показывает активную policy |
| `i2c <dev> policy show rules` | показывает policy, whitelist и blacklist целиком |
| `i2c <dev> policy show whitelist` | показывает только whitelist |
| `i2c <dev> policy show blacklist` | показывает только blacklist |
| `i2c <dev> policy set whitelist` | делает `whitelist` активной policy |
| `i2c <dev> policy set blacklist` | делает `blacklist` активной policy |
| `i2c <dev> policy whitelist add <reg> <val>` | добавляет пару в whitelist |
| `i2c <dev> policy whitelist del <reg> <val>` | удаляет пару из whitelist |
| `i2c <dev> policy whitelist clear` | очищает весь whitelist |
| `i2c <dev> policy blacklist add <reg> <val>` | добавляет пару в blacklist |
| `i2c <dev> policy blacklist del <reg> <val>` | удаляет пару из blacklist |
| `i2c <dev> policy blacklist clear` | очищает весь blacklist |

Например, если нужно посмотреть текущие правила `axp15060` и удалить пару `{ reg:0x14 val:0x17 }` из whitelist, последовательность будет такой:

```text
i2c axp15060 policy show rules
i2c axp15060 policy whitelist del 0x14 0x17
i2c axp15060 policy show whitelist
```

Отдельная команда `save` для `i2c` больше не нужна. Изменения policy и списков правил сохраняются автоматически в `flash:/config/i2c/<device>.json` сразу после успешной команды.

## `reboot`, `quit` и поведение закрытия

`quit` и `exit` работают просто: сессия печатает `Bye!` и закрывается. `reboot` требует явного подтверждения. В текущем коде принимаются формы `reboot -y [delay_ms]` и `reboot confirm [delay_ms]`. Без подтверждения консоль вернёт сообщение об ошибке и ничего не сделает.

Пример:

```text
reboot -y 1000
```

После этого консоль печатает `OK (rebooting)` и закрывает сессию.

## Что удобно автоматизировать, а что нет

Консоль вполне подходит для простых инженерных automation-сценариев, но только если помнить о её природе.

| Сценарий | Насколько это удобно через TCP-консоль |
|---|---|
| ручная диагностика и разовый осмотр | очень удобно |
| файловые операции и проверка конфигов | удобно |
| сетевые правки и reboot | удобно, но с риском разрыва сессии |
| строгая бинарная автоматизация | лучше использовать DCP2 |
| bulk file transfer | часто удобнее HTTP `/flash`, `/sd` или `/tar/...` |

Если нужен формальный контракт, двоичные статусы и события, консоль уже не лучший инструмент. Для этого есть DCP2. Если нужно быстро руками посмотреть, что происходит на устройстве прямо сейчас, TCP-консоль почти всегда остаётся самым коротким путём.

## Что проверять при проблемах с консолью

Когда консоль не отвечает или ведёт себя странно, полезно начать с нескольких простых проверок.

| Симптом | Что проверить первым |
|---|---|
| соединение на `8888` не устанавливается | поднялась ли сеть и правильный ли IP у устройства |
| после `ip addr set ...` консоль пропала | переподключиться уже к новому адресу |
| файловые команды отвечают `FS not ready` | дождаться монтирования `sd:/` или `flash:/` |
| `smi` даёт неожиданные результаты | помнить, что SMI не входит в текущий стандартный startup path |
| tab-completion ведёт себя ограниченно | это нормально, completion реализован только для части команд и аргументов |

Если нужен самый короткий рабочий сценарий для начала, он сейчас такой: подключиться на `8888`, выполнить `help`, затем `ip addr show`, потом `cd flash`, `ls /`, `ls /config` и уже после этого переходить к более узким командам `i2c`, `spi`, `tar` или сетевым изменениям.
