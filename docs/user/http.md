# HTTP-сервер (порт 80)

HTTP в `bvstk` это не “большой web backend”, а компактная служебная поверхность поверх `lwIP sockets`. Она нужна для трёх разных пользовательских сценариев: получить JSON-состояние через `/api/*`, передать файлы через `/sd`, `/flash` и `/tar/...`, либо открыть web UI, который раздаётся из `flash:/www/`.

Сервер работает просто: на одно TCP-соединение приходится один HTTP-запрос, после ответа соединение закрывается. В ответах используется `HTTP/1.0` и `Connection: close`, поэтому клиенту не стоит рассчитывать на keep-alive или долгоживущую HTTP-сессию.

## Как устроен HTTP снаружи

Если смотреть на HTTP как пользователь, у него есть три слоя, и их полезно не путать.

| Поверхность | Для чего она нужна |
|---|---|
| `/api/*` | JSON-операции и получение состояния устройства |
| `/sd/...`, `/flash/...`, `/tar/...` | передача отдельных файлов и каталогов |
| всё остальное через `GET` | раздача web UI из `flash:/www/` |

Именно в таком порядке сервер и обрабатывает запросы. Сначала он пытается распознать JSON API, затем файловые маршруты, и только потом воспринимает запрос как попытку открыть web-страницу или статический ресурс из `flash:/www/`.

## Что важно помнить про ответы

По пользовательскому поведению сервер сейчас довольно предсказуем. JSON-маршруты отвечают `application/json; charset=utf-8`, текстовые ошибки приходят как `text/plain`, а файлы и tar-потоки отдаются отдельными специализированными ответами. Для `PUT`-запросов сервер требует либо `Content-Length`, либо `Transfer-Encoding: chunked`. Если тело передаётся без одного из этих механизмов, запрос получает `411 Length Required`.

На практике это означает, что `curl` и обычные HTTP-клиенты подходят хорошо, а экзотические клиенты, которые пытаются писать тело запроса без стандартного framing, будут работать плохо.

## JSON API

Пользовательский JSON API здесь намеренно плоский. У него нет авторизации, сессий и сложной объектной модели. Это сделано не потому, что это “публичный REST API”, а наоборот: потому что он рассчитан на доверенный инженерный контур и на простые автоматизированные сценарии.

### Что можно прочитать через `GET`

| Маршрут | Что возвращает |
|---|---|
| `GET /api/version` | дату и время сборки, HTTP-порт |
| `GET /api/rtos` | uptime, tick rate, текущее и минимальное free heap |
| `GET /api/net` | текущее состояние сети и MAC |
| `GET /api/fs` | готовность и размер томов `flash` и `sd` |
| `GET /api/qspi` | состояние QSPI FS и параметры её окна во флеше |
| `GET /api/i2c` | список I2C-устройств из `config_store` |
| `GET /api/i2c?name=<device>` | подробный конфиг одного I2C-устройства |

Для быстрой проверки прошивки обычно достаточно нескольких простых вызовов:

```sh
curl http://<device-ip>/api/version
curl http://<device-ip>/api/net
curl http://<device-ip>/api/fs
curl http://<device-ip>/api/i2c
```

Это даёт неплохую первую картину: жива ли HTTP-служба вообще, поднята ли сеть, готов ли `flash:/`, и дошёл ли `config_store` до загрузки I2C-конфигов.

### Что можно менять через `PUT`

| Маршрут | Что делает |
|---|---|
| `PUT /api/net` | сохраняет сетевую конфигурацию и при желании применяет её на лету |
| `PUT /api/i2c` | обновляет часть конфига одного I2C-устройства |
| `PUT /api/reboot` | инициирует reboot через watchdog |

`PUT /api/net` принимает IP, маску или prefix, gateway, MAC и флаг `apply`. Если `apply=true`, настройки не только сохраняются в `config_store`, но и сразу переносятся в текущий `netif`. Для пользователя это значит, что HTTP-сессия может пережить изменение сетевых параметров не всегда, особенно если вы меняете IP-адрес устройства прямо из текущего соединения.

Пример типичного запроса:

```sh
curl -X PUT \
  -H 'Content-Type: application/json' \
  --data '{"ip":"192.168.1.10/24","gateway":"192.168.1.1","mac":"02:12:34:56:78:9a","apply":true}' \
  http://<device-ip>/api/net
```

`PUT /api/i2c` рассчитан не на создание новых устройств, а на изменение уже существующего конфига, который пришёл из `config_store`. В запросе обязательно должно быть `name`, а менять можно policy, autopoll и списки правил. Адрес устройства и имя файла через этот API не меняются.

Типичный пример:

```sh
curl -X PUT \
  -H 'Content-Type: application/json' \
  --data '{"name":"rtc","policy":"whitelist","autopoll_enabled":true,"autopoll_regs":[0,1,2]}' \
  http://<device-ip>/api/i2c
```

`PUT /api/reboot` требует явного подтверждения `confirm:true`. Это намеренно сделано как защита от случайной перезагрузки.

```sh
curl -X PUT \
  -H 'Content-Type: application/json' \
  --data '{"confirm":true,"delay_ms":1000}' \
  http://<device-ip>/api/reboot
```

## Диагностические `diag`-эндпоинты

Отдельный класс HTTP-маршрутов предназначен не для обычной эксплуатации, а для инженерной диагностики. Это `PUT /api/diag/i2c/read`, `PUT /api/diag/i2c/write`, `PUT /api/diag/smi/read`, `PUT /api/diag/smi/write`, `PUT /api/diag/mem/read` и `PUT /api/diag/mem/write`.

Их полезно воспринимать как прямой доступ к живым шинам и памяти, а не как удобный “безопасный API”. В этом и их ценность, и их риск.

| Маршрут | Практический смысл |
|---|---|
| `PUT /api/diag/i2c/read` | прочитать регистр I2C-устройства по имени или адресу |
| `PUT /api/diag/i2c/write` | записать I2C-регистр |
| `PUT /api/diag/smi/read` | прочитать MDIO/SMI-регистр |
| `PUT /api/diag/smi/write` | записать MDIO/SMI-регистр |
| `PUT /api/diag/mem/read` | прочитать память или MMIO по адресу |
| `PUT /api/diag/mem/write` | записать память или MMIO по адресу, только с `confirm:true` |

Несколько практических оговорок здесь важнее любых примеров. I2C-диагностика реально полезна в текущей системе, потому что I2C-подсистема стартует штатно. SMI-маршруты в коде есть, но сама SMI-подсистема в стандартном `main()` сейчас не запускается, поэтому успешность этих операций зависит от конкретного runtime-состояния и не должна восприниматься как гарантированная часть обычного user flow. Маршруты `mem/read` и `mem/write` особенно опасны, потому что работают с сырыми адресами и не вводят отдельного policy-layer поверх самого HTTP API.

Пример I2C-чтения:

```sh
curl -X PUT \
  -H 'Content-Type: application/json' \
  --data '{"name":"axp15060","reg":19}' \
  http://<device-ip>/api/diag/i2c/read
```

Пример I2C-записи:

```sh
curl -X PUT \
  -H 'Content-Type: application/json' \
  --data '{"name":"axp15060","reg":19,"val":18}' \
  http://<device-ip>/api/diag/i2c/write
```

Пример прямой записи в память:

```sh
curl -X PUT \
  -H 'Content-Type: application/json' \
  --data '{"addr":1073741824,"val":1,"confirm":true}' \
  http://<device-ip>/api/diag/mem/write
```

Такие операции стоит делать только тогда, когда вы точно понимаете, какой блок железа стоит за этим адресом или регистром.

## Файловый HTTP API

HTTP умеет не только отдавать JSON, но и работать с файлами на `sd:/` и `flash:/`. Это отдельная поверхность, полезная тогда, когда нужно передать файл без захода в shell.

| Маршрут | Во что маппится |
|---|---|
| `GET/PUT /sd/<path>` | `sd:/<path>` |
| `GET/PUT /flash/<path>` | `flash:/<path>` |
| `GET/PUT /tar/sd/<dir>` | tar-поток каталога `sd:/<dir>` |
| `GET/PUT /tar/flash/<dir>` | tar-поток каталога `flash:/<dir>` |

Обычный файловый `GET` отдаёт конкретный файл как бинарный поток. Если путь указывает на каталог, сервер отвечает `404 Not Found`, потому что directory listing через HTTP здесь не предусмотрен. Обычный файловый `PUT` создаёт или полностью перезаписывает файл, но не создаёт промежуточные каталоги. Если нужного каталога ещё нет, запись закончится ошибкой `open failed`.

Примеры:

```sh
curl -o network.json http://<device-ip>/flash/config/network.json
curl -T network.json http://<device-ip>/flash/config/network.json
```

Tar-режим нужен, когда надо передать целое дерево файлов, например web UI или комплект конфигов. `GET /tar/...` упаковывает каталог в tar-поток, а `PUT /tar/...` распаковывает tar в каталог назначения. Это самый удобный путь для массовой загрузки содержимого `flash:/www/`.

```sh
curl -o www.tar http://<device-ip>/tar/flash/www
curl -T www.tar http://<device-ip>/tar/flash/www
```

С пользовательской точки зрения это один из самых полезных HTTP-сценариев в проекте: shell хорош для ручного осмотра, а tar-over-HTTP удобен для быстрой доставки дерева файлов на устройство.

## Web UI

Если запрос не попал ни в `/api/*`, ни в файловые маршруты, сервер пытается отдать статический ресурс из `flash:/www/`. Это и есть web UI текущей прошивки.

Поведение здесь простое. Запрос на `/` ведёт к `flash:/www/index.html`. Запрос на путь с завершающим `/` ведёт к `index.html` внутри соответствующего подкаталога. Query string и fragment на выбор файла не влияют.

| URL | Что обычно ищется на устройстве |
|---|---|
| `/` | `flash:/www/index.html` |
| `/ui/` | `flash:/www/ui/index.html` |
| `/assets/app.js` | `flash:/www/assets/app.js` |

Тип контента определяется по расширению файла. HTML, CSS, JS, JSON, PNG, JPEG, SVG, ICO и TXT сейчас имеют отдельные MIME-типы, всё остальное отдаётся как `application/octet-stream`.

Практический смысл этого простой: если UI не открывается, сначала нужно проверять не роутинг браузера, а само наличие правильных файлов в `flash:/www/`.

## Когда идти в HTTP, а когда нет

Пользовательски HTTP полезен, когда нужна одна из трёх вещей: быстро получить JSON-состояние, передать файл, либо открыть web UI. Но это не лучший интерфейс для всего подряд.

| Задача | Что обычно удобнее |
|---|---|
| посмотреть живое состояние и быстро походить по файлам | TCP shell |
| получить структурированный статус или обновить сетевой/I2C-конфиг | HTTP `/api/*` |
| массово загрузить или выгрузить дерево файлов | HTTP `/tar/...` |
| наблюдать события и binary control plane | DCP2 |

Если задача упирается в асинхронные события, бинарные статусы или наблюдение за действиями shell и host-логики, HTTP уже не лучший инструмент. Там полезнее DCP2. Если нужна ручная диагностика и навигация по файловой системе, shell обычно быстрее и понятнее.

## Что стоит проверять при проблемах

Когда HTTP “не работает”, чаще всего проблема лежит не в самом сервере как таковом, а в одном из нижележащих слоёв.

| Симптом | Что проверить первым |
|---|---|
| `curl` не подключается | поднялась ли сеть и правильный ли IP у устройства |
| `/api/i2c` отвечает странно или пусто | готов ли `config_store` |
| запись в `/flash/...` не проходит | готов ли `flash:/` и существует ли каталог назначения |
| web UI отдаёт `404` | действительно ли нужные файлы лежат в `flash:/www/` |
| `diag/smi/*` ведёт себя непредсказуемо | помнить, что SMI сейчас не входит в стандартный startup path |

Для быстрой первичной проверки обычно достаточно этой последовательности:

```sh
curl http://<device-ip>/api/version
curl http://<device-ip>/api/net
curl http://<device-ip>/api/fs
curl http://<device-ip>/
```

Если эти четыре шага дают осмысленные ответы, значит сеть, HTTP-слой, как минимум часть `flash:/` и базовый web-root уже живы. Дальше имеет смысл переходить к более узким JSON или файловым маршрутам под конкретную задачу.
