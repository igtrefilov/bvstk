# Руководство пользователя

Этот документ предназначен для человека, который работает с уже собранной и загруженной системой. Здесь нет глубокого разбора кода и внутренней архитектуры FreeRTOS. Вместо этого объясняется, как подойти к устройству как к рабочему инструменту: как подключиться, что проверить первым делом, какими интерфейсами пользоваться и в каких случаях лучше идти в TCP-консоль, а в каких — в HTTP или DCP2.

Если вам нужна не эксплуатация, а разработка, переходите в `../dev/guide.md`. Если нужен более подробный материал по конкретной подсистеме, используйте профильные документы из `docs/user`, `docs/dev` и `docs/reference`.

## Что пользователь видит после загрузки

После успешной прошивки и старта система обычно выглядит как сетевое устройство с несколькими сервисами управления. У неё есть IP-адрес, текстовая TCP-консоль на порту `8888`, DCP2-сервер на `8889` и HTTP-сервер на `80`. Кроме того, внутри доступны два файловых тома: SD-карта как `sd:/` и участок QSPI-флеша как `flash:/`.

Самая частая ошибка на первом шаге — воспринимать систему как “ещё один web-сервер”. На самом деле web-интерфейс здесь только одна из оболочек. Основной диагностический инструмент для человека — TCP-консоль. Основной структурированный интерфейс для внешнего клиента — DCP2. HTTP удобен там, где нужен UI или простой автоматизированный JSON/file API.

## С чего начать после включения

Обычный пользовательский сценарий лучше строить так.

Сначала проверьте, что устройство поднялось по сети. Если IP неизвестен, ориентируйтесь на настройки вашей сети и дефолтные адреса, которые описаны в полном reference. Когда IP найден, имеет смысл сразу открыть TCP-консоль: это самый быстрый способ понять текущее состояние системы.

Подключение выглядит так:

```sh
telnet <device-ip> 8888
```

или, если `telnet` не установлен:

```sh
nc <device-ip> 8888
```

После подключения удобно выполнить несколько базовых команд:

```text
help
ip addr show
pwd
ls
```

Этого достаточно, чтобы понять три вещи: отвечает ли shell, поднята ли сеть с ожидаемым адресом, и доступны ли файловые системы.

## Когда использовать TCP-консоль

TCP-консоль — основной интерфейс ручного управления. Она удобна, когда вы хотите быстро проверить состояние, посмотреть конфиг, вручную прочитать или записать регистр, сменить сетевой адрес или перекинуть файлы между `sd:/` и `flash:/`.

Самая сильная сторона консоли в том, что она даёт мгновенную обратную связь в человекочитаемом виде. Для диагностики это почти всегда лучший первый шаг. Вы не обязаны помнить внутреннюю архитектуру проекта, чтобы сделать базовую проверку. Достаточно понимать набор ключевых утилит:

- `ip` для сети
- `fs`-подобные команды для файлов и каталогов
- `i2c` для I2C-устройств
- `smi` для MDIO/PHY
- `mem` для низкоуровневой диагностики MMIO/BRAM

При этом нужно помнить, что shell имеет доступ к потенциально опасным операциям. Команды `mem`, прямые записи в регистры и изменение сетевых параметров могут ломать текущую сессию или переводить устройство в состояние, из которого потом придётся восстанавливаться вручную.

## Как работать с файлами

Для пользователя файловая модель проста: `sd:/` соответствует SD-карте, а `flash:/` — файловой системе на QSPI. Обычно `flash:/` используют для конфигурации и web-ресурсов, а `sd:/` — для обмена файлами или временного хранения.

Чаще всего нужны четыре операции:

1. посмотреть, что есть на томе
2. перейти в каталог
3. прочитать или скопировать конфиг
4. перенести набор файлов между `sd:/` и `flash:/`

Минимальный набор команд:

```text
ls flash:/config
cd flash:/config
cat network.json
cp sd:/cfg.json flash:/config/cfg.json
```

Если вы обновляете web-ресурсы или набор конфигов, удобнее думать не “о Linux shell”, а о небольшой встроенной сервисной файловой системе. Лучше делать изменения явно, проверяя путь и том, чем рассчитывать на сложные shell-паттерны.

## Как работать с сетью

Большинство пользовательских операций начинаются именно с сети, потому что все основные интерфейсы работают поверх неё. Если вы меняете IP, это почти всегда оборвёт текущие TCP-соединения. Это нормальное поведение, а не признак сбоя.

Чтобы посмотреть текущие настройки:

```text
ip addr show
ip route show
ip link show
```

Чтобы поменять IP и маршрут:

```text
ip addr set 192.168.0.10/24
ip route set default via 192.168.0.1
```

После этого будьте готовы переподключиться уже по новому адресу. Если сеть настроена неверно и устройство “пропало”, дальше придётся идти либо через JTAG/UART, либо через известный fallback-сценарий из полного reference-документа.

## Как использовать HTTP

HTTP здесь нужен не как основной интерактивный интерфейс, а как удобная транспортная оболочка. Через него удобно:

- читать и сохранять JSON-конфиги
- дергать диагностические API
- загружать или скачивать файлы
- отдавать web UI из `flash:/www/`

Если задача звучит как “открыть интерфейс в браузере” или “сделать автоматизацию простым HTTP-запросом”, это почти наверняка зона HTTP. Если задача звучит как “хочу понять, почему не работает” или “надо быстро вручную проверить состояние”, лучше идти в TCP-консоль.

## Как использовать DCP2

DCP2 нужен в тех случаях, когда внешний клиент должен работать с устройством не через текстовый shell и не через HTTP, а через стабильный бинарный протокол. Это особенно полезно для интеграции с внешним ПО, где важны строгие статусы, формальные типы сообщений и асинхронные события.

Практически DCP2 полезен в двух режимах.

Первый режим — обычный request/response, когда клиент хочет выполнить `PING`, `MEM`, `I2C` или `SMI` операцию и получить нормализованный бинарный ответ.

Второй режим — подписка на асинхронные события через `NOTIFY`. Именно этот режим позволяет видеть попытки записи в регистры, успешные изменения, denied-события и действия, пришедшие не только от DCP-клиента, но и, например, от telnet shell или host.

Полная формальная спецификация находится в `../dcp2.md`. Для повседневной эксплуатации удобнее не читать спецификацию вручную, а использовать готовый скрипт `scripts/dcp2/monitor_notify.py`.

## Мониторинг событий через DCP2

Самый удобный пользовательский сценарий для DCP2 сейчас такой: открыть shell в одном окне и монитор событий в другом. Это даёт очень наглядную картину, когда вы экспериментируете с регистрами или хотите видеть реакцию системы на внешние транзакции.

Подключение монитором:

```sh
./scripts/dcp2/monitor_notify.py <device-ip> --port 8889
```

Только I2C-события:

```sh
./scripts/dcp2/monitor_notify.py <device-ip> --port 8889 --buses i2c
```

Только telnet и host как источники:

```sh
./scripts/dcp2/monitor_notify.py <device-ip> --port 8889 --sources telnet,host
```

После этого в shell можно выполнить что-то вроде:

```text
i2c axp15060 w 0x13 0x12
```

и увидеть, как shell отчитался об операции, а DCP2-клиент получил `NOTIFY_EVENT`. Для пользователя это один из самых понятных способов проверить, что события о регистровых транзакциях действительно уходят наружу.

## Как работать с I2C и SMI

Для оператора здесь важна не внутренняя архитектура PL-ядер, а практическое правило: любые регистровые операции проходят через политики доступа. Это значит, что запись в регистр может быть не только успешной, но и сознательно отклонённой конфигурацией устройства.

Типичные команды для I2C:

```text
i2c list
i2c axp15060 info
i2c axp15060 policy show rules
i2c axp15060 r 0x13
i2c axp15060 w 0x13 0x12
```

Если нужно отредактировать правила доступа, это делается через подкоманды `policy`. Например, чтобы удалить из whitelist пару `{ reg:0x14 val:0x17 }`, используется:

```text
i2c axp15060 policy whitelist del 0x14 0x17
```

Отдельный `i2c ... save` для этого не нужен: успешные изменения policy и rule lists сохраняются автоматически.

Типичные команды для SMI:

```text
smi list
smi @1 info
smi @1 r 0
smi @1 w 0 0x8000
```

Если запись не проходит, это не обязательно аппаратная проблема. Сначала нужно проверить политику устройства, список разрешённых или запрещённых регистров и то, был ли конфиг вообще загружен из `flash:/config`.

## Что делать осторожно

Система даёт достаточно низкоуровневый доступ, поэтому часть операций стоит рассматривать как сервисные или аварийные.

Особой аккуратности требуют:

- команды `mem r/w`
- ручные записи в регистры питания и PHY
- изменение сетевых параметров во время активной сессии
- обновление файлов в `flash:/config/`
- перезапись web UI в `flash:/www/`

Эти операции не запрещены, но их лучше делать осознанно, понимая контекст. Если вы не уверены, какой именно эффект будет у записи в регистр, лучше сначала посмотреть состояние, политику и наличие уведомлений в DCP2.

## Типовой рабочий маршрут пользователя

Если сформулировать нормальный “человеческий” порядок работы, он выглядит так.

Сначала вы убеждаетесь, что устройство живо по сети. Затем заходите в TCP-консоль и проверяете адрес, файловые системы и базовые подсистемы. Если задача связана с конфигами, работаете с `flash:/config`. Если задача связана с интеграцией или наблюдением за внешними транзакциями, открываете рядом DCP2-монитор. Если нужен UI или простой скриптовый API, переходите к HTTP.

Именно в таком порядке документация и выстроена: сначала shell, затем HTTP и DCP2, а полная reference-детализация остаётся отдельным слоем.
