# Сеть

Сетевая модель текущей прошивки `bvstk` проста: устройство поднимает один Ethernet-интерфейс `eth0` на PS GEM, использует статический IPv4 и берёт адрес, маску, gateway и MAC либо из `config_store`, либо из встроенных дефолтов. Для пользователя здесь важнее всего не детали `lwIP`, а то, что сеть стартует достаточно рано, живёт отдельно от shell и HTTP, и опирается на файл `flash:/config/network.json`.

Ключевой практический факт: в текущем startup path устройство не запускает DHCP-клиент и не получает адрес автоматически от DHCP-сервера. Если конфиг сети не был загружен, применяется fallback static-конфигурация.

## Что происходит при старте

После запуска `config_store` прошивка создаёт сетевой поток `lan_thrd`. Он ждёт готовность `config_store` примерно до 5 секунд, пытается забрать из него MAC и IPv4-параметры, затем вызывает `lwip_init()`, создаёт `netif`, поднимает его и запускает input-thread Ethernet-драйвера.

Если к этому моменту в `config_store` уже есть полный сетевой конфиг, устройство использует именно его. Если нет, прошивка не уходит в DHCP, а поднимает интерфейс на встроенных значениях.

| Параметр | Значение по умолчанию |
|---|---|
| IP | `192.168.0.10` |
| netmask | `255.255.255.0` |
| gateway | `192.168.0.1` |
| MAC | `00:0a:35:00:01:02` |

Для пользователя это означает очень полезную вещь: даже если `flash:/config/network.json` отсутствует, повреждён или не успел загрузиться, устройство всё равно обычно можно искать по известному fallback-адресу `192.168.0.10`.

## Где хранится сетевой конфиг

Постоянная конфигурация сети хранится в `flash:/config/network.json`. Именно этот файл читает `config_store`, и именно в него попадают изменения, сохранённые через shell-команду `ip` или через HTTP `PUT /api/net`.

Практически это означает, что у пользователя есть два разных состояния сети.

| Состояние | Что это значит |
|---|---|
| runtime-состояние | то, что уже применено к живому `netif` |
| сохранённый конфиг | то, что лежит в `flash:/config/network.json` и переживёт reboot |

В нормальном случае эти состояния совпадают, но это не обязано быть так всегда. Можно, например, заменить файл `network.json` без немедленного применения, либо наоборот, применить настройки на живую сеть и потерять доступ к старому адресу раньше, чем вы успеете переподключиться.

## Формат `network.json`

Текущая реализация ожидает обычный JSON с IPv4-полями и MAC-адресом. Он выглядит так:

```json
{
  "ipv4": {
    "ip": "192.168.0.10",
    "netmask": "255.255.255.0",
    "gateway": "192.168.0.1"
  },
  "mac": "00:0a:35:00:01:02"
}
```

Для пользователя здесь важно не переусложнять модель. Прошивка не хранит отдельный режим `dhcp` против `static`, не держит hostname и не опирается на расширенную сетевую схему. Нужны только `ip`, `netmask`, `gateway` и `mac`.

## Работа через TCP shell

Самый прямой пользовательский интерфейс для сети это shell на порту `8888`. Команды `ip` позволяют посмотреть текущее runtime-состояние, поменять адресацию и сохранить её в `flash:/config/network.json`.

| Команда | Что делает |
|---|---|
| `ip addr show` | показывает текущий IP, prefix, netmask и gateway |
| `ip link show` | показывает MAC |
| `ip route show` | показывает default gateway |
| `ip addr set <IPv4>/<prefix>` | меняет IP и маску, сразу применяет и сохраняет |
| `ip route set default via <gw>` | меняет gateway, сразу применяет и сохраняет |
| `ip link set address <mac>` | меняет MAC, сразу применяет и сохраняет |
| `ip save` | сохраняет текущее runtime-состояние `netif` в `flash:/config/network.json` |

Типичный сценарий выглядит так:

```text
ip addr show
ip addr set 192.168.0.20/24
ip route set default via 192.168.0.1
ip link set address 00:0a:35:00:01:02
```

Практически важно, что команды `ip addr set`, `ip route set ...` и `ip link set address ...` в текущем коде не только применяют параметры на лету, но и сразу пытаются сохранить их в `flash:/config/network.json`. Поэтому отдельный `ip save` нужен в основном тогда, когда вы хотите просто зафиксировать уже существующее runtime-состояние `netif`, а не когда вы меняете параметры через сами `ip ... set`.

## Работа через HTTP

Тот же пользовательский смысл доступен и через `PUT /api/net`. Этот маршрут принимает JSON с IP, netmask или prefix, gateway, MAC и флагом `apply`.

| Поле | Для чего нужно |
|---|---|
| `ip` | IPv4-адрес, либо в виде `a.b.c.d/prefix`, либо отдельно |
| `netmask` | маска, если `ip` задан без `/prefix` |
| `prefix` | альтернатива `netmask` |
| `gateway` | default gateway |
| `mac` | MAC-адрес |
| `apply` | применять ли изменения к живому `netif` сразу |

Если `apply=true`, устройство не только сохраняет конфиг, но и немедленно переносит его в текущий интерфейс. Если `apply=false`, изменения остаются только в `config_store` и применятся уже после следующего старта или после отдельного ручного применения.

Пример немедленного применения:

```sh
curl -X PUT \
  -H 'Content-Type: application/json' \
  --data '{"ip":"192.168.0.20/24","gateway":"192.168.0.1","mac":"00:0a:35:00:01:02","apply":true}' \
  http://<old-ip>/api/net
```

Пример “только сохранить”:

```sh
curl -X PUT \
  -H 'Content-Type: application/json' \
  --data '{"ip":"192.168.0.20/24","gateway":"192.168.0.1","mac":"00:0a:35:00:01:02","apply":false}' \
  http://<device-ip>/api/net
```

Для пользователя здесь есть важная оговорка. Даже если HTTP-ответ успел вернуться как `200 OK`, изменение IP с `apply=true` может тут же оборвать текущие TCP-соединения, потому что устройство уже переехало на новый адрес.

## Как безопасно менять IP

Любая смена IP на живом устройстве почти неизбежно рвёт текущую shell- или HTTP-сессию. Это не ошибка. Это нормальное следствие того, что вы меняете сам адрес, по которому разговариваете с устройством.

Обычно безопаснее всего действовать так: сначала понять, на какой новый адрес вы переводите устройство, затем применить изменение, а потом сразу переподключаться уже к новому IP.

```text
ip addr set 192.168.0.20/24
ip route set default via 192.168.0.1
```

После этого текущая сессия может отвалиться. Дальше уже нужно заходить по новому адресу:

```sh
telnet 192.168.0.20 8888
curl http://192.168.0.20/api/net
```

Если вы меняете параметры через HTTP, логика та же самая. Запрос посылается на старый адрес, а следующий запрос уже должен идти на новый.

## Как восстановить доступ, если адрес потерян

Когда устройство перестало отвечать после сетевых изменений, полезно начинать не с предположения “сеть сломалась”, а с нескольких простых гипотез.

| Ситуация | Что проверять первым |
|---|---|
| устройство перестало отвечать сразу после смены IP | попробовать новый адрес, а не старый |
| непонятно, на какой адрес оно переехало | посмотреть `arp -a`, `ip neigh` или просканировать подсеть |
| конфиг кажется повреждённым | проверить `flash:/config/network.json` через shell, SD или файловый HTTP API |
| вообще нет уверенности, какой конфиг применился | помнить про fallback на `192.168.0.10` |

Практически полезны такие шаги:

```sh
arp -a
ip neigh
nmap -sn 192.168.0.0/24
```

После нахождения IP уже имеет смысл проверить порт `8888` или `80`. Если доступа по сети нет совсем, следующим шагом обычно становится запуск через JTAG, вход в TCP shell и ручная правка `flash:/config/network.json` или новая установка параметров через `ip ...`.

## Что показывают пользовательские интерфейсы

В shell команды `ip addr show`, `ip route show` и `ip link show` показывают именно текущее runtime-состояние интерфейса. HTTP `GET /api/net` тоже показывает текущее состояние `netif`, а не просто содержимое файла `network.json`.

Это полезно помнить, если вы сравниваете “что реально поднято” с “что лежит в конфиге”. Для сохранённого состояния лучше отдельно смотреть сам файл `flash:/config/network.json`, а для живого состояния доверять shell и `GET /api/net`.

## Чего в текущей сетевой модели нет

Чтобы не строить неверные ожидания, полезно явно зафиксировать несколько ограничений текущей реализации.

| Ожидание | Что на самом деле |
|---|---|
| устройство само получит адрес по DHCP | нет, в текущем startup path используется статический IPv4 |
| можно сохранить только часть конфигурации без остальных полей | практически сеть живёт как полный набор `ip/netmask/gateway/mac` |
| изменение IP через shell или HTTP не повлияет на текущую сессию | повлияет, соединение может оборваться сразу |
| `GET /api/net` показывает только сохранённый конфиг | нет, он показывает текущее состояние интерфейса |

Если нужен самый короткий рабочий рецепт, он сейчас такой: сначала ищите устройство на `192.168.0.10`, проверяйте `ip addr show` или `GET /api/net`, меняйте адрес осознанно, и помните, что окончательная правда о persistent-настройках лежит в `flash:/config/network.json`.
