# Web UI

В текущем проекте web UI это не отдельное frontend-приложение со сборочным пайплайном, а набор готовых статических файлов, которые хранятся в репозитории в `web/assets/` и на устройстве оказываются в `flash:/www/`. HTTP-сервер потом раздаёт их как обычную статику. Для пользователя это важнее всего понимать именно так: UI здесь живёт как содержимое встроенной файловой системы, а не как особый сервис со своей базой, роутером или отдельным backend-процессом.

Практически это означает две простые вещи. Во-первых, если UI не работает, одна из первых точек проверки это наличие правильных файлов в `flash:/www/`. Во-вторых, обновление UI это в основном задача доставки статики на устройство, а не “деплоя приложения” в привычном web-смысле.

## Откуда берётся UI

Исходным пользовательским набором файлов в текущем репозитории является каталог `web/assets/`. В нём уже лежит готовая статика: `index.html`, CSS, JavaScript и изображения. Никакого обязательного `npm build`, Vite, webpack или другого frontend-build step для текущего user flow здесь нет.

| Где лежит | Что это означает |
|---|---|
| `web/assets/` | локальная копия готовой статики на хосте |
| `flash:/www/` | та же статика после загрузки на устройство |
| `http://<device-ip>/` | HTTP-раздача содержимого `flash:/www/` |

Для пользователя это полезное упрощение. Если вы видите нужный файл в `web/assets/` и успешно загрузили его в `flash:/www/`, то дальше HTTP-сервер должен уметь его отдать без дополнительной серверной сборки.

## Как UI раздаётся на устройстве

HTTP-сервер в текущей прошивке использует `flash:/www/` как web-root. Запрос на `/` приводит к `flash:/www/index.html`. Запросы на подкаталоги с завершающим `/` приводят к `index.html` внутри соответствующего подкаталога. Всё остальное интерпретируется как запрос конкретного статического файла.

Это даёт очень прямую модель:

| URL | Что ищется на устройстве |
|---|---|
| `/` | `flash:/www/index.html` |
| `/ui/` | `flash:/www/ui/index.html` |
| `/css/app.css` | `flash:/www/css/app.css` |
| `/js/main.js` | `flash:/www/js/main.js` |

Именно поэтому при проблемах с UI полезно думать не “сломался браузерный роутинг”, а “существует ли нужный файл в `flash:/www/` и отдает ли его HTTP вообще”.

## Что обычно должно быть в `web/assets/`

В текущем проекте каталог `web/assets/` уже используется как готовая структура UI. Пользовательски полезно воспринимать её как обычное дерево статических файлов, а не как набор исходников для отдельного toolchain.

| Тип содержимого | Типичный пример |
|---|---|
| точка входа | `web/assets/index.html` |
| стили | `web/assets/css/...` |
| JavaScript | `web/assets/js/...` |
| изображения | `web/assets/logo.png`, `web/assets/bg.png` |

Практически важное требование здесь одно: внутри HTML и связанных файлов лучше использовать относительные пути, чтобы тот же набор ресурсов одинаково работал и после загрузки в `flash:/www/`, и при локальной проверке содержимого.

## Рекомендуемый способ загрузки

Текущий проект уже содержит штатный загрузчик UI: `web/upload_flash_www.sh` и `web/upload_flash_www.py`. Именно этот путь стоит считать нормальным. Он не просто отправляет файлы по HTTP, а сначала создаёт нужные каталоги через TCP-консоль, потом загружает файлы через HTTP `PUT`, а поверх этого ведёт `manifest.json`, чтобы в следующие разы передавать только изменившиеся файлы.

Это делает его заметно удобнее ручного `curl` по одному файлу.

Быстрый запуск выглядит так:

```sh
./web/upload_flash_www.sh <device-ip>
```

Тот же сценарий напрямую через Python:

```sh
python3 ./web/upload_flash_www.py <device-ip> --src ./web/assets
```

С практической точки зрения этот flow сейчас лучший по трём причинам. Он работает с уже существующей моделью `flash:/www/`, не требует вручную создавать каталоги, и умеет пропускать неизменившиеся файлы на основе хэшей.

## Что делает `manifest.json`

При штатной загрузке в `flash:/www/` создаётся и обновляется `manifest.json`. Это не “часть UI как интерфейса”, а служебный файл загрузчика. Он содержит список файлов, их размеры и SHA-256, чтобы следующая загрузка могла понять, что реально изменилось.

| Файл | Роль |
|---|---|
| `flash:/www/manifest.json` | служебная карта хэшей и размеров для incremental upload |

Для пользователя это означает простую вещь: если вы пользуетесь `upload_flash_www.py`, повторные обновления UI обычно будут заметно быстрее, чем первый полный upload.

## Полезные опции штатного загрузчика

У `upload_flash_www.py` есть несколько опций, которые реально полезны в повседневной работе.

| Опция | Когда полезна |
|---|---|
| `--dry-run` | посмотреть, что будет загружено, ничего не меняя |
| `--force` | загрузить всё, игнорируя существующий manifest |
| `--no-mkdir` | пропустить создание каталогов через TCP-консоль |
| `--http-port` | если HTTP слушает не на `80` |
| `--console-port` | если TCP-консоль слушает не на `8888` |
| `--dst` | если нужно сменить целевой путь на устройстве |
| `--http-prefix` | если нужен нестандартный HTTP-путь для `PUT` |
| `--manifest` | если нужен другой путь или имя manifest-файла |

Если нужен безопасный первый шаг перед реальной загрузкой, `--dry-run` почти всегда лучший выбор.

```sh
python3 ./web/upload_flash_www.py <device-ip> --src ./web/assets --dry-run
```

## Альтернативный способ: tar через HTTP

Если нужен не incremental upload, а перенос всего дерева за одну операцию, текущий проект поддерживает и другой нормальный путь: tar-over-HTTP. Для этого можно собрать tar из `web/assets/` и отправить его в `/tar/flash/www`.

```sh
tar -C web/assets -cf www.tar .
curl -T www.tar http://<device-ip>/tar/flash/www
```

Этот способ особенно полезен, когда нужно быстро положить на устройство целиком новый набор статики без участия Python-загрузчика. Но у него нет hash-based пропуска неизменившихся файлов, поэтому как регулярный workflow он обычно менее удобен, чем `upload_flash_www.py`.

## Ручная загрузка

Ручной путь тоже возможен, но это уже скорее режим отладки, чем основной рабочий сценарий. Можно создать каталог через консоль и положить туда файлы либо с SD, либо отдельными `HTTP PUT`.

Например:

```text
mkdir flash:/www
cp -r sd:/www flash:/www
```

или

```sh
curl -T web/assets/index.html http://<device-ip>/flash/www/index.html
```

Практически неудобство ручного пути в том, что вам самому приходится следить за иерархией каталогов, а HTTP `PUT` отдельных файлов не создаёт промежуточные директории автоматически.

## Как проверять, что UI действительно загрузился

После загрузки UI нет смысла ограничиваться фактом “скрипт не упал”. Полезно проверить и файловый слой, и HTTP-раздачу.

| Проверка | Что она подтверждает |
|---|---|
| `ls flash:/www` в TCP-консоли | файлы действительно лежат на устройстве |
| `curl http://<device-ip>/` | HTTP отдаёт `index.html` |
| `curl http://<device-ip>/api/version` | JSON API жив и UI сможет обращаться к backend-маршрутам |
| `GET /api/qspi` или `GET /api/fs` | `flash:/` вообще готов и доступен |

Обычно хороший минимальный набор такой:

```sh
curl http://<device-ip>/
curl http://<device-ip>/api/version
curl http://<device-ip>/api/fs
```

Если эти шаги проходят, значит и web-root, и HTTP, и как минимум часть backend-поверхности уже работают.

## Что делать, если UI не открывается

Когда UI не открывается, полезно разделять три разных класса проблем: нет файлов, HTTP не отдаёт файлы, или сама страница загрузилась, но не может работать с API.

| Симптом | Что проверять первым |
|---|---|
| браузер получает `404` на `/` | существует ли `flash:/www/index.html` |
| `/` открывается, но часть ресурсов не грузится | лежат ли нужные CSS/JS/изображения по ожидаемым путям |
| UI открывается, но “пустой” или не показывает состояние | отвечает ли `/api/version`, `/api/net`, `/api/fs` и другие нужные API |
| загрузчик не может записать файлы | доступна ли сеть, готов ли `flash:/`, работает ли TCP-консоль на `8888` |
| после обновления видна старая версия | действительно ли новые файлы попали в `flash:/www/`, а не осталась старая статика |

Практически почти всегда стоит начинать с двух вопросов: “что реально лежит в `flash:/www/`?” и “что реально отдаёт `curl http://<device-ip>/`?”.

## Как лучше думать о UI в этом проекте

В текущем `bvstk` web UI это одна из оболочек над системой, а не центральный интерфейс, вокруг которого построено всё остальное. Основной инженерный контроль по-прежнему проще и надёжнее делать через TCP-консоль и DCP2. UI полезен там, где нужен браузерный доступ, обзор состояния и удобная человеческая точка входа.

Поэтому и документационно к нему лучше относиться как к набору ресурсов в `flash:/www/`, которые можно обновлять, проверять и резервировать как обычные файлы. В этом проекте это самая точная и практически полезная модель.
