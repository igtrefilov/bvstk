# bvstk

`bvstk` — это прошивка для целевой Zynq-платформы семейства `Burevestnik`. Она поднимает FreeRTOS и lwIP на PS-части, монтирует SD и QSPI-файловые системы, загружает конфигурацию из JSON и даёт несколько каналов управления устройством: текстовую TCP-консоль, HTTP API и бинарный протокол DCP2. Отдельная важная часть проекта — связка с кастомными PL-ядрами для `I2C`, `SMI/MDIO` и `SPI`.

Этот репозиторий больше не рассматривается как один огромный справочник. Документация разделена по ролям: отдельно есть материалы для пользователя системы, отдельно для разработчика, и отдельно сохранён полный reference-документ, в который вынесён прежний объёмный README. Такой формат проще читать, проще поддерживать и проще расширять без превращения входной страницы в машинно сгенерированный каталог.

## Что это за система

На практике `bvstk` играет роль control plane для платы на базе Zynq. После загрузки система поднимает сеть, монтирует локальные тома, читает конфигурацию и предоставляет несколько точек входа для оператора или внешнего клиента. С одной стороны, это “прикладная” прошивка с сетевыми сервисами. С другой, это слой управления над кастомным аппаратным дизайном, где в PL реализованы ядра для доступа к внешним шинам и периферии.

Важно, что проект нельзя рассматривать в отрыве от аппаратной платформы. В соседнем hardware-репозитории лежит Vivado-проект `Burevestnik_21`, top-level `SystemVerilog`, экспортированные `bit/xsa` и локальный `ip_repo` с кастомными IP-блоками. Прошивка использует именно эти HW-артефакты при сборке платформы Vitis, а адреса MMIO, прерывания и состав периферии должны совпадать с тем, что описано в `xsa`.

## Система в одном абзаце

Нормальный жизненный цикл такой. Аппаратный проект собирается в Vivado и экспортируется как `bit/xsa`. Затем `bvstk` собирается в Vitis на основе этого `xsa`, загружается по JTAG и после старта поднимает три основных интерфейса управления: TCP-консоль на `8888`, DCP2-сервер на `8889` и HTTP-сервер на `80`. Через эти интерфейсы оператор или клиент может работать с файлами, сетью, конфигурацией, регистрами I2C/SMI и диагностикой системы.

## Быстрый старт

Если вам нужен минимальный путь “собрать, залить, проверить, что всё живо”, он выглядит так:

1. Убедитесь, что у вас есть актуальные `design.xsa` и `design.bit`, соответствующие целевому Vivado-проекту.
2. Соберите прошивку:

```sh
cd <repo-root>
./scripts/vitis/build.sh
```

3. Загрузите bitstream и ELF по JTAG:

```sh
./scripts/vitis/run_jtag.sh
```

4. После старта системы подключитесь к устройству:

```sh
telnet <device-ip> 8888
```

5. Для проверки DCP2 и асинхронных событий используйте:

```sh
./scripts/dcp2/monitor_notify.py <device-ip> --port 8889
```

Если нужен нормальный пошаговый маршрут с пояснениями, а не краткий список команд, читайте `docs/user/guide.md` и `docs/dev/guide.md`.

## Доступные интерфейсы после старта

После загрузки прошивки доступны три основных канала взаимодействия:

| Интерфейс | Порт | Для чего используется |
|---|---:|---|
| TCP-консоль | `8888` | ручное управление, shell-подобные команды, диагностика |
| DCP2 | `8889` | внешний бинарный клиент, request/response и async `event` |
| HTTP | `80` | web UI, JSON API, файловые операции |

Эти интерфейсы решают разные задачи. TCP-консоль удобна для человека, когда нужно быстро проверить состояние системы или вручную выполнить команду. HTTP подходит для UI и прикладной интеграции на уровне REST-подобных запросов. DCP2 нужен там, где важна жёсткая структура обмена, стабильный бинарный формат и асинхронные уведомления, например через сервис `NOTIFY`.

## Как связаны `hw_platform/fpga` и `bvstk`

Эти два репозитория образуют одну систему.

hardware platform отвечает за аппаратный дизайн:
- Vivado-проект `Burevestnik_21`
- top-level `src_vivado/Burevestnik_top.sv`
- экспорт `Burevestnik_top.bit` и `Burevestnik_top.xsa`
- локальный `ip_repo` с кастомными PL-ядрами `I2C`, `SMI`, `SPI`

репозиторий `bvstk` отвечает за программную часть:
- Vitis/XSCT build
- FreeRTOS/lwIP/FatFs
- загрузку и сохранение конфигурации
- TCP/HTTP/DCP2 сервисы
- логику поверх PL-ядер

Для разработчика это означает простое, но жёсткое правило: прошивка и hardware platform должны двигаться согласованно. Если в Vivado меняются адреса, IRQ, BRAM layout или состав IP-ядер, прошивка может потребовать соответствующих изменений в коде и конфигурации сборки.

Подробно эта связка разобрана в `docs/dev/hardware-platform.md`.
